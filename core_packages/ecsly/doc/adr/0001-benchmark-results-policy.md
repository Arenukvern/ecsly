# ADR 0001: Benchmark Results Policy

## Status

Accepted.

## Context

`ecsly` is performance-oriented, but benchmark numbers are easy to misuse.
Single machine results can become stale, hide bottlenecks, or imply guarantees
that the package does not actually provide.

We need benchmark output that is useful for maintainers and honest for readers:

- reproducible commands;
- machine-readable output for tracking over time;
- environment metadata next to every result;
- enough scenarios to show both strong paths and bottlenecks;
- conservative README claims.

## Decision

Benchmark results are release artifacts. They must be generated by the benchmark
runner, not hand-written.

The canonical publishable local report command is median-of-5:

```sh
dart run benchmark/run.dart --limits --samples=5 \
  --markdown-out=benchmark/results/latest.md \
  --json-out=benchmark/results/latest.json
```

For automation that only needs JSON:

```sh
dart run benchmark/run.dart --json --limits --out benchmark/results/latest.json
```

Every published report must include:

- generated timestamp;
- command used;
- OS and Dart SDK version;
- processor count;
- git commit and dirty-tree status when available;
- metric name, category, entity count, raw value, derived `ns/op`, and notes.

The README may link to benchmark reports, but it should not present benchmark
numbers as universal claims. Hardware, OS, Dart SDK, runtime mode, and package
version all affect results.

Scorecards describe movement, not causes. A report can say that throughput,
latency, RSS, or cache-miss metrics changed. It cannot claim the cause without a
matching CPU/allocation/GC/timeline profile sidecar. Causal notes should cite
profile evidence and name the hot symbol or phase observed there, such as
`QueryCache.getOrCompute`, `CommandQueue.execute`, archetype migration, raw
query chunk iteration, or accidental hot-path `List`/`Map`/closure allocation.

Thresholds are local regression checks, not public guarantees. The historical
scalar YAML form remains a higher-is-better minimum for throughput metrics:

```yaml
query_iteration_ops_per_sec: 10000000
```

Lower-is-better latency and memory metrics use explicit ceilings:

```yaml
game_frame_20k_p95_micros:
  max: 5000
  lowerIsBetter: true
```

Profile sidecars should live under `benchmark/results/profiles/` and use a
filename that includes the scorecard date or commit plus the investigated
metric, for example `2026-07-01-query-cache-miss.cpu-profile.json`.

## Benchmark Shape

The default benchmark suite covers:

- typed-column query iteration;
- raw chunk query iteration;
- query count/any paths;
- mutable typed-column iteration;
- object-component command flush;
- archetype migration;
- query-cache hit and forced miss;
- typed-data event send/read;
- RSS delta across repeated frame-style updates.

The optional `--limits` mode adds entity-count scaling measurements so we can
observe where throughput bends as data size grows.

## Consequences

Positive:

- Results are reproducible and reviewable.
- JSON output can feed dashboards or regression tooling.
- Markdown output can be published directly.
- Strong paths and bottlenecks are visible in the same report.

Trade-offs:

- Benchmark reports are machine-specific.
- Limit scans take longer than default regression checks.
- Thresholds remain separate from published results; passing CI does not imply a
  particular user-machine throughput.

## Non-goals

- Claiming universal FPS/entity-count guarantees.
- Comparing against other ECS libraries without a dedicated fairness plan.
- Replacing application-level profiling.
