# ecsly DX FAQ

Practical usage notes for the published `ecsly` core package. This FAQ keeps
runtime usage separate from companion app, Flutter, codegen, and agent-only
guidance.

## Quickstart

```dart
CREATE_WORLD: final world = World()
ENTITIES: world.entities.create()
RESOURCES: world.resources.push/get<T>()
RESOURCE_DEFAULT: world.addResourceIfAbsent<T>(() => T())

SCHEDULES: world.createSchedule('Update')
TIME: world.upsertResource(ScheduleTimeResource(...))
```

## API Temperature Lanes

```dart
HOT: schedules + query/queryMut/queryExt + typed columns
WARM: commands + flush + resources + prepareQuery + query revisions
COLD: package:ecsly_app actions/drafts/domain-id projections
HOST: package:ecsly_flutter scope/controller/selectors/loops
```

Keep persistent app ids as component data. Runtime `Entity` handles are not
save-file or route identities. If a UI repeatedly starts from a route id, use a
cold projection in `ecsly_app`/plugin code, not a core `World` id API.

App-layer and Flutter host packages live in this public repo under
`core_packages/ecsly_app` and `core_packages/ecsly_flutter`. Their source has
moved; hosted pub.dev publication is tracked separately.

Core exposes `queryRevision` / `structuralRevision` for topology changes that
affect archetype membership and query caches. Treat them as topology epochs, not
operation counters: one command flush that applies any number of spawn, despawn,
add-component, or remove-component changes advances the epoch once. Query-cache
eviction is structural membership eviction only. It is not component/resource
change tracking, and direct in-place writes through query/queryMut do not bump
these revisions.

Use `world.addResourceIfAbsent<T>(() => T(...))` for cold-path default
resources installed by plugins or bootstraps. Use `world.upsertResource(...)`
when replacing/updating resource state is intended.

Use `world.addPluginIfAbsent(plugin)` only for optional bootstrap/dependency
installation. Use strict `world.addPlugin(plugin)` when duplicate installation
should fail, and use `removePlugin(name)` followed by `addPlugin(plugin)` for
runtime replacement.

Flutter/app hosts that need selective rebuilds should use app-layer invalidation
batches from `package:ecsly_app`/`package:ecsly_flutter`; hot ECS worlds do not
pay for retained entity/type summary objects.

## 🧭 Quickstart FAQ

**Q: What is the smallest way to start?**  
A: Create a `World`, register component types, spawn or mutate entities, then
query the world.

```dart
final world = World();

world.components.registerObjectComponent<CounterComponent>();

final entity = world.reserveEmptyEntity().entity;
world.spawnBundle(entity, ComponentBundle.fromLists([CounterComponent(1)]));
world.flush();

for (final (_, counter) in world.query<CounterComponent>()) {
  print('counter=${counter.value}');
}
```

For cold-path app/tooling code, `world.spawnComponents([...])` is a shorthand
for reserving an entity and spawning one component bundle:

```dart
final entity = world.spawnComponents([
  TodoId('todo-1'),
  TodoTitle('Slice app state into components'),
]);
world.flush();
```

**Q: Why do examples call `flush()`?**  
A: Spawn, insert, remove, and despawn are structural changes. They are queued so
iteration remains deterministic and safe. Call `world.flush()` when queued
changes must be visible before the next lookup or query.

**Q: How do I mutate an object component?**  
A: Use `queryMut<T>()` for mutable object-component iteration, or get a mutable
entity wrapper.

```dart
for (final (_, counter) in world.queryMut<CounterComponent>()) {
  counter.value += 1;
}
```

**Q: How do I run logic through a schedule?**  
A: Add a system function to a named schedule, run the schedule, then flush at the
frame boundary if systems queued structural changes.

```dart
world.createSchedule('Update').add((world) {
  for (final (_, counter) in world.queryMut<CounterComponent>()) {
    counter.value += 1;
  }
});

world.runSchedule('Update');
world.flush();
```

**Q: Which examples should I read first?**  
A: Start with `example/basic_world.dart` for object components. Then read
`example/scheduled_run.dart` for an extension component backed by typed column
storage plus a cold object component for debug metadata.

Next examples:

- `example/extension_component.dart` - marker component plus typed facade.
- `example/commands_and_resources.dart` - deferred commands and global resources.
- `example/simd_columns.dart` - stride-4 typed column and SIMD view.

## Codegen Policy

`ecsly_codegen` is **separate code-generation tooling**, not part of `ecsly` runtime.
Use it in packages that define marker components for generated column/facade classes, and keep generated artifacts out of runtime-only imports.

- `ecsly` (`core_packages/ecsly`) stays runtime-focused and does not depend on `ecsly_codegen`.
- Component packages or test fixtures add `ecsly_codegen` (and `build_runner` tooling) where `@EcsComponent` annotations are used.
- Import path for the runtime package is `package:ecsly/ecsly.dart`.
- Full workflow (annotate -> `build_runner` -> `registerExtension`) belongs to
  the companion `ecsly_codegen` package docs when that package is available in
  your workspace or on pub.dev.

## 🧪 Example Component Choices

**Q: Why do core examples avoid `PositionComponent` and `VelocityComponent`?**  
A: Those are gameplay-space concepts owned by domain packages or applications.
Core examples stay domain-neutral so they teach the runtime contract without
encouraging duplicate, incompatible movement components.

**Q: When should I use an object component?**  
A: Use `registerObjectComponent<T>()` when data is cold, low-volume, easier to
model as a normal Dart object, or still being explored. Object components are
the easiest way to learn the ECS flow.

**Q: When should I use an extension component?**  
A: Use `registerExtension<TComponent, TFacade>()` when data is compact, numeric,
frequently queried, or present on many entities. Extension facades access
typed-column storage without creating one Dart object per entity per hot path.

**Q: Why does `queryExt<EnergyComponent, Energy>()` need two types?**  
A: `EnergyComponent` is the component identity stored in archetype signatures.
`Energy` is the facade used to read/write packed column data. This keeps runtime
component lookup explicit while giving systems nice typed fields.

```dart
for (final (_, energy) in world.queryExt<EnergyComponent, Energy>()) {
  energy.current += energy.regenPerTick;
}
```

**Q: How do commands differ from direct mutation?**  
A: Direct mutation changes existing component data. Commands queue structural
changes such as spawn, despawn, insert, and remove so iteration can finish
safely. Call `world.flush()` to apply them.

```dart
for (final (entity, counter) in world.query<CounterComponent>()) {
  if (counter.value >= 2) entity.remove<CounterComponent>();
}
world.flush();
```

**Q: How do I find a component when I only have app data?**  
A: Model persistent app ids as normal components or plugin data. Core `Entity`
handles are runtime pointers, not durable identity. When host/app code needs
repeated selected-id or route-id lookup, keep that id-to-entity projection in
the app/plugin layer instead of `ecsly` core.

```dart
class TodoId extends Component {
  const TodoId(this.value);
  final String value;
}
```

Use `maybeGetComponent<T>(entity)` for nullable direct entity reads when you
already have an `Entity`. Systems that run per frame should use query/raw chunk
APIs.

**Q: What are resources for?**  
A: Resources are global singleton state: frame clocks, settings, caches, queues,
or simulation-wide counters. They avoid inventing a special entity just to hold
global data.

Resources may be immutable snapshots, mutable runtime state, or a small mix of
both depending on purpose. Keep resources data-only: put behavior in systems,
not callbacks, command queues, system registration, or domain logic inside the
resource.

**Q: When is a resource too big?**  
A: If a resource stores repeated user/domain records, a large
`Map<Entity, DomainState>`, per-item loading state, validation state, or business
rules, split it. Put repeated state on entities as components and keep the
resource as a small index, selected-id set, filter, queue, status table, or cache.

**Q: What is the app/UI rule of thumb?**  
A: User-visible records and independently changing slices usually become
components. World/session state becomes resources.

| State shape | Prefer |
| --- | --- |
| Todo item, coffee card, content block, unit, form section | Component(s) |
| Dirty flag, load request, per-item error, selection marker | Component/tag |
| Current route, active filter, selected id set, viewport | Resource |
| Action/sync status table, outbox, cache, metrics | Resource |
| Spawn/despawn/add/remove request | `EcsCommand` / command queue |

```dart
world.upsertResource(FrameClockResource(deltaSeconds: 1 / 60));

void tickFrameClockSystem(World world) {
  world.getResource<FrameClockResource>().frame += 1;
}

void queueSpawnOrRemoveSystem(World world) {
  for (final (entity, counter) in world.query<CounterComponent>()) {
    if (counter.value >= 2) entity.remove<CounterComponent>();
  }
}
```

## 🚀 Schedule Launchpad

```dart
CREATE: world.createSchedule('Update')
ADD: schedule.add(systemFunction)
CHAIN: schedule.add(sys1).then(sys2)
JOB: schedule.addJobSystem(certifiedJobSystem, name: 'collision')
PARALLEL: schedule.parallel([sys1, sys2])
RUN: world.runSchedule('Update')
RUN_ASYNC: await world.runScheduleAsync('Update')
FLUSH: add flushAllWithTimingSystem at end
TIME_SOURCE: ScheduleTimeResource(deltaSeconds: dt, elapsedSeconds: t)
POLICY: world.getResource<ScheduleExecutionPolicyResource>()..mode = ScheduleExecutionPolicy.deterministic..workerCount = 4
```

`schedule.parallel(...)` is still shared-world async work. Use `addJobSystem(...)`
plus `runScheduleAsync(...)` when a hot path has a certified extract/partition/
execute/merge implementation and needs deterministic worker execution.

## 📈 Benchmark Lab

```dart
RUN: dart run benchmark/run.dart
CHECK: dart run benchmark/run.dart --check
SCORECARD: dart run benchmark/run.dart --limits --samples=5
THRESHOLDS: benchmark/thresholds.yaml
```

`--check` compares one current sample with `benchmark/thresholds.yaml`. Treat it
as a smoke gate for fast local feedback, not a publishable benchmark scorecard.
Existing scalar entries are higher-is-better floors. Nested entries with `max`
and `lowerIsBetter: true` are lower-is-better ceilings for `micros` and memory
diagnostics. A line like `REGRESSION query_count_ops_per_sec: actual=...` means
the metric crossed its local threshold; it does not explain the cause. Read
metric units before drawing conclusions: `ops/sec` is throughput and higher is
better, `micros` is frame/latency cost and lower is better, and RSS/allocation
signals are coarse memory diagnostics until a profiler confirms the hot symbol.

Current command names:

```sh
dart run benchmark/run.dart
dart run benchmark/run.dart --check
dart run benchmark/run.dart --limits --samples=5 --markdown-out=benchmark/results/latest.md --json-out=benchmark/results/latest.json
dart run benchmark/run.dart --markdown --limits --out benchmark/results/latest.md
dart run benchmark/run.dart --json --limits --out benchmark/results/latest.json
```

`query_iteration`, `raw_query_chunk`, `mutable_query_iteration`, and
`frame_rss_delta` are the typed-column/SoA headline metrics.
`command_flush` remains a separate object-component diagnostic until typed-data
upsert extractors are benchmarked on the same path.

Use the README baseline as a historical anchor, not as a universal target.
Generated reports should be median-of-5 (`--samples=5`) when making claims,
with raw samples preserved in `benchmark/results/latest.json`; that JSON file is
the publishable local scorecard. Use single runs and `--check` only for fast
local smell checks.

`benchmark/profile_commands.dart` emits JSON proof with top-level `scenarios`
and `schemaId: ecsly.core.command_profile` exactly, with no `.v1` suffix. It
records environment, git commit/dirty state when available, the requested
sample count, per-scenario runs, and raw phase timing samples. Use this JSON to
pick a phase before opening DevTools.

When a scorecard metric regresses or improves unexpectedly, switch to Dart
profile tooling and save the profile as a sidecar under
`benchmark/results/profiles/`:

```sh
dart --observe=0 --pause-isolates-on-start run benchmark/run.dart --json
```

Open the DevTools URL printed by `dart --observe`, resume the paused isolate,
and capture CPU, allocation, GC, and timeline evidence. Inspect
`CommandQueue.execute`, `QueryCache.getOrCompute`, query/raw chunk iterators,
archetype migration, structural revision bumps, cache invalidation, and
accidental `List`/`Map`/closure/record allocations. Benchmark deltas say what
moved; profile captures explain why.

Architecture improvement candidates should be judged in this order:

- Preserve hot SoA iteration and avoid per-entity heap work.
- Keep structural revision updates coalesced around flush/migration boundaries.
- Keep query-cache invalidation internal and topology-driven.
- Improve `query_count`/`query_any` without adding host-observation state.
- Move UI/app invalidation evidence to `ecsly_app`/`ecsly_flutter`, not core.

## 🧩 Plugin Marketplace

```dart
ADD: world.addPlugin(PluginClass())
FEATURE: MyFeaturePlugin() // registers package-owned components/systems
```

Domain plugins own their own component vocabulary. Core `ecsly` provides the
plugin hook, component registration, schedules, resources, and commands; it does
not own game-specific components such as position, velocity, or health.

## 🎯 SIMD Accelerator

```dart
final column = archetype.getColumn(id) as FloatColumn;
final simd = column.simdView; // Only if stride % 4 == 0
if (simd != null) {
  final scale = Float32x4.splat(1.1);
  for (int i = 0; i < simd.length; i++) {
    simd[i] = simd[i] * scale;
  }
}
```

For a runnable version, see `example/simd_columns.dart`. SIMD works best when
the component uses a `FloatColumn(stride: 4)` so each entity row maps to one
`Float32x4` lane.

## 🚨 Safety Zone

- ✅ Use extension types for hot paths
- ❌ Avoid class components in tight loops (GC)
- ❌ Deprecated + forbidden for web interop: `dart:html`, `dart:js_util` (`js:utils`)
- ✅ Required web interop path: `dart:js_interop` (plus `dart:js_interop_unsafe` only when needed)
- ✅ Flush at schedule end
- ✅ SIMD when stride % 4 == 0

**Memory Hook:** "World → Entity → Query → Component → Schedule → Events"
