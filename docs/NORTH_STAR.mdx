# North Star

`ecsly` exists to become the public canonical home for publishable Dart and
Flutter ECS packages, with an honest path toward a practical Dart game-engine
stack.

The repo should make entity/component architecture usable from plain Dart,
Flutter apps, tools, tests, and eventually game loops without pretending that
future engine work already exists.

## What This Repo Owns

| Surface | Owned here |
| --- | --- |
| Core runtime | `core_packages/ecsly`: pure Dart entities, components, archetypes, resources, commands, events, systems, schedules, queries, and benchmarks. |
| App workflow layer | `core_packages/ecsly_app`: app actions, drafts, invalidation, identity projections, and cold lookup patterns. |
| Code generation | `core_packages/ecsly_codegen`: typed-column factory annotations and builders. |
| Flutter bridge | `core_packages/ecsly_flutter`: scope, controller, selectors, actions, and loop widgets for Flutter hosts. |
| Public stewardship | Repository-level docs, release automation, Steward actions, contributor paths, and package validation loops. |

## What This Repo Does Not Own

| Boundary | Rule |
| --- | --- |
| Private or internal workspaces | Work outside this repository is not a public contract until it is migrated here. |
| Complete game engine claims | The North Star points toward a game-engine stack; the current repo is an experimental package family. |
| Host app identity | Product-specific navigation, branding, storage policy, and domain models belong in consuming apps. |
| Compatibility mirrors | Do not add bidirectional sync scripts or hidden mirror machinery. |
| Absent packages and tools | Do not document packages, plugins, or validation tracks that are not present here as shipped capabilities. |

## Values

- Keep public claims tied to code, tests, examples, releases, or explicit blocked
  state.
- Keep core `ecsly` pure Dart and runtime-focused.
- Let companion packages own app, Flutter, and generation concerns rather than
  pushing host responsibilities into the core runtime.
- Prefer small, verifiable layers over broad architecture promises.
- Treat plans as temporary; extract durable decisions into ADRs or FAQs and then
  remove stale planning artifacts.

## Extension Model

`ecsly` may grow by adding package-owned sub-stars when a surface becomes large
enough to need its own center. A sub-star must name its parent relationship to
this global North Star and must not change the global ownership boundary by
itself.

Good sub-star candidates:

- `core_packages/ecsly/NORTH_STAR.md` for the pure Dart runtime.
- `core_packages/ecsly_flutter/NORTH_STAR.md` if Flutter host behavior becomes
  broad enough to need a separate direction.
- A package DESIGN FAQ section when the boundary is still small.

## Decision Gate

Before durable structural changes, classify North Star impact:

| Impact | Route |
| --- | --- |
| `none` | Use the native package workflow. |
| `applies` | Proceed and link the relevant package docs. |
| `clarifies` | Update this North Star, `docs/DESIGN_FAQ.mdx`, `docs/DX_FAQ.mdx`, or the package FAQ. |
| `sub_star` | Add or update a package sub-star with an explicit parent boundary. |
| `amends` | Write an ADR and update this North Star after the decision is accepted. |
| `conflicts` | Stop until the maintainer accepts an ADR and North Star diff. |

## Proof Rule

Steward checks prove routing and contract shape. Package behavior is proven by
native Dart/Flutter validation, publish dry-runs, examples, runtime evidence, or
published package state. Keep those proof types separate in public docs.
